────────────────────────────────────────────────────────────────────────────────
 1. DATABASE MIGRATION (JSON/CSV → SQLite or Postgres)
    Effort: Medium | Impact: CRITICAL
────────────────────────────────────────────────────────────────────────────────

THE PROBLEM:
Everything — call logs, leads, agent configs, workflows — lives in flat JSON
and CSV files. In production, this means race conditions if two calls end
simultaneously, zero query capability, no pagination, and data loss risk.

WHAT TO DO:
Replace data/call_logs.json, data/leads.csv, and data/workflows.json with a
proper database. SQLite (via Prisma or Drizzle) for a single-server deploy,
or Postgres/Supabase if you want multi-instance scaling later. The
data/agent_config.json bridge can stay as-is since it's a hot-reload config
pattern, not a data store.

IMPACT:
Unlocks real CRM search/filter, call history pagination, concurrent write
safety, and analytics queries.

FILES AFFECTED:
- analytics.py (writes call_logs.json + leads.csv)
- dashboard/app/api/leads/route.ts (reads leads.csv)
- dashboard/app/api/recordings/route.ts (reads call_logs.json)
- dashboard/app/api/agent-config/route.ts (reads/writes agent_config.json)
- dashboard/components/LeadsCRM.tsx (displays lead data)
- data/ directory (replaced by DB)


────────────────────────────────────────────────────────────────────────────────
 2. AUTHENTICATION & API ROUTE PROTECTION
    Effort: Medium | Impact: CRITICAL
────────────────────────────────────────────────────────────────────────────────

THE PROBLEM:
The dashboard and all its API routes (/api/dispatch, /api/agent-config,
/api/leads) are completely open. Anyone with the URL can configure your agents,
trigger calls, and view all lead data.

WHAT TO DO:
Add NextAuth.js or Clerk. Protect every API route with session middleware. Add
role-based access (admin can configure agents, viewer can only see logs).
Rate-limit the dispatch endpoint so nobody can spam 1000 calls.

IMPACT:
Non-negotiable for deployment. Without this, the product is a security
liability.

FILES AFFECTED:
- dashboard/app/api/* (all API routes need auth middleware)
- dashboard/app/layout.tsx (wrap with auth provider)
- dashboard/components/Sidebar.tsx (show user info, logout)
- NEW: dashboard/middleware.ts (NextAuth/Clerk middleware)
- NEW: dashboard/app/login/page.tsx (login page)


────────────────────────────────────────────────────────────────────────────────
 3. PROVIDER FAILOVER CHAINS (TTS/STT/LLM)
    Effort: Low-Medium | Impact: HIGH
────────────────────────────────────────────────────────────────────────────────

THE PROBLEM:
If Sarvam's API goes down mid-call or Groq has a blip, the call just dies.
There's no graceful degradation. The _build_tts() and _build_llm() functions
pick one provider and commit to it for the entire call.

WHAT TO DO:
Implement a fallback chain in the agent code:
  - TTS:  Sarvam → Deepgram Aura → OpenAI TTS (automatic failover)
  - LLM:  Groq → Gemini → OpenAI (with timeout-based switching)
  - STT:  Deepgram Nova-2 is already solid, but add a health-check ping on
          agent startup.
Log every failover event so you can track provider reliability over time.

IMPACT:
Turns "demo that sometimes breaks" into "reliable product that self-heals."

FILES AFFECTED:
- agent_outbound.py (_build_tts, _build_llm, _build_stt)
- agent_inbound.py  (_build_tts, _build_llm, _build_stt)
- config_outbound.py (add fallback chain config variables)
- config_inbound.py  (add fallback chain config variables)
- analytics.py (log failover events)


────────────────────────────────────────────────────────────────────────────────
 4. LIVE CALL MONITORING PANEL
    Effort: High | Impact: HIGH (key differentiator)
────────────────────────────────────────────────────────────────────────────────

THE PROBLEM:
Once a call is dispatched, you have zero visibility. You can't see if it
connected, how long it's been going, what the AI is saying, or if it's stuck.
You only find out after the call ends (via logs).

WHAT TO DO:
Add a "Live Calls" section to the dashboard that:
  - Shows all active LiveKit rooms with call duration, caller number, and
    agent status
  - Streams the real-time transcript (LiveKit's data channels or a WebSocket
    from the agent)
  - Lets an admin "listen in" (join the LiveKit room as a silent participant)
  - Shows a live sentiment indicator (positive/neutral/negative) based on
    the conversation

IMPACT:
This is the single biggest "wow factor" feature for anyone evaluating the
product. It's also operationally critical — you need to know when calls are
going wrong *while they're happening*.

FILES AFFECTED:
- NEW: dashboard/app/live/page.tsx (live monitoring page)
- NEW: dashboard/components/LiveCallCard.tsx (per-call status card)
- NEW: dashboard/app/api/live-calls/route.ts (fetches active rooms from LiveKit API)
- agent_outbound.py (emit transcript events via LiveKit data channel)
- agent_inbound.py  (emit transcript events via LiveKit data channel)
- dashboard/components/Sidebar.tsx (add "Live Calls" nav item)


────────────────────────────────────────────────────────────────────────────────
 5. HEALTH CHECKS, STARTUP VALIDATION & GRACEFUL SHUTDOWN
    Effort: Low | Impact: HIGH
────────────────────────────────────────────────────────────────────────────────

THE PROBLEM:
run.py blindly spawns agents and hopes they work. If a critical env var is
missing, the agent crashes on the first call (not at startup). There's no
/health endpoint for load balancers. Docker has no readiness probe.

WHAT TO DO:
  - Startup validation: On agent boot, verify all API keys are valid (ping
    each provider), verify the SIP trunk exists in LiveKit, verify the config
    JSON is parseable. Fail fast with clear error messages.
  - Health endpoint: Add a GET /health route to both the dashboard and the
    Python agents that returns provider connectivity status, active call
    count, and uptime.
  - Graceful shutdown: On SIGTERM, finish any active calls (don't just kill
    them), flush pending analytics, then exit.
  - Docker Compose: A proper docker-compose.yml that runs both agents + the
    dashboard with health checks, restart policies, and volume mounts for
    the data directory.

IMPACT:
Makes the difference between "I deploy and pray" and "I deploy and monitor."

FILES AFFECTED:
- run.py (add startup validation, graceful shutdown)
- agent_outbound.py (add health check server, signal handlers)
- agent_inbound.py  (add health check server, signal handlers)
- NEW: dashboard/app/api/health/route.ts (dashboard health endpoint)
- Dockerfile (add HEALTHCHECK instruction)
- docker-compose.yml (add health checks, restart policies, volumes)
